}

Now we'll add a Girlfriend to the synchronizer

 

Problem is, Girlfriend's are really hard to create!

@Test def canAddGirlfriendToEmptySynchronizer() {
    // Given a new Synchronizer
    def o = new ConcurrentGirlfriendSynchronizer()
    // And a Girlfriend to add.
    // PLEASE DO NOT CHANGE config.xml OR THIS WILL BREAK!!
    // None of this setup is relevant to what we're testing.
    def fac = GirlfriendFactory.newInstance("config.xml")
    def props = new GirlfriendProperties(
           hair: Color.BROWN, eyes: Color.BROWN, age: [18..40])
    def gf = fac.newGirlfriend(props)

    // When we add a Girlfriend...
    o.addGirlfriend(gf)

    // Then there is one Girlfriend in the synchronizer...
    assertThat(o.size(), is(1))
    // And the right Girlfriend is in the synchronizer
    assertThat(o.getGirlfriends(), contains(gf))
What's the problem?
  • Getting a Girlfriend to test with is hard
  • Setup of test distracts from the intent of the test
  • Fragile - imagine if config.xml is changed, it might break things
Some possible fixes?
  • Manually create a stub Girlfriend. What if it's a big interface? What if it changes?
  • Create a Mock for it? But all we need a Stub, and Mocks aren't stubs.
  • What else could there possibly be (and it might fix the triangulation thing too)?
Ipsedixit can provide
Auto Triangulation and Lightweight Stubs

Let's rewrite that "name" test using Ipsedixit.

 

class ConcurrentGirlfriendSynchronizer {
@Test def canSetName() {
    // Given a new object
    def o = new ConcurrentGirlfriendSynchronizer()
    // And a name to use...
    def name = TestData.randomString()

    // When the value of the 'name' property to some value
    o.setName(name)

    // Then the value of that property should be the same
    // as what it was initially set to
    assertThat(o.getName(), is(name))
}

Ipsedixit gives you Automatic Triangulation by randomising values.

  • If you don't really care what the value is, let Ipsedixit make it up for you
  • No two test runs will run the same - hard coding values won't work
  • TestData is Ipsedixit's main utility class. It has a bunch of handy methods

Let's rewrite that "add Girlfriend" test using Ipsedixit.

 

@Test def canAddGirlfriendToEmptySynchronizer() {
    // Given a new Synchronizer
    def o = new ConcurrentGirlfriendSynchronizer()
    // And a Girlfriend to add
    def gf = TestData.stub(Girlfriend.class)

    // When we add the Girlfriend...
    o.addGirlfriend(gf)

    // Then there is one Girlfriend in the synchronizer...
    assertThat(o.size(), is(1))
    // And the right Girlfriend is in the synchronizer
    assertThat(o.getGirlfriends(), contains(gf))
}

By introducing Ipsedixit, we've simplified creation of Stubs

  • Ipsedixit can give you stubs for Interfaces and non-final Classes
  • Tests are a lot easier to read. No messy and distracting setup.
  • Stubs implement equals, toString and hashCode
  • Stubs will throw exceptions for any other method invocation

Ipsedixit can help you test.
  • You want some value but you don't care what it is
  • You want a stub object that is strictly not a mock
  • You want some utility methods to select values from a List or Array, provide whitespace, inline stubs and more
